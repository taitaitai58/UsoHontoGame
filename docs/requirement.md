## ウソホントゲーム運営Webアプリケーション 要件定義書

**文書番号:** PM-20251105-001
**作成日:** 2025年11月5日
**作成者:** (あなたの一流のPM/SE)

### 1\. 概要

#### 1.1. プロジェクトの目的

本プロジェクトの目的は、チームビルディングや懇親会等のイベントで利用する「ウソホントゲーム（Two Truths and a Lie）」を、円滑かつインタラクティブに運営するためのWebアプリケーションを開発することである。
手動での集計や進行に伴うオペレーション負荷を軽減し、参加者全員がゲームに集中できる環境を提供すると同時に、リアルタイムのスコア表示によってゲームの盛り上がりを最大化する。

#### 1.2. システムの対象範囲

本システムは、ゲームの事前準備（チーム分け、エピソード登録）から、ゲーム中の進行管理（出題、回答、集計）、最終結果の発表まで、ゲーム運営に関わる一連のプロセスを対象とする。

### 2\. 利用者（ロール）定義

本システムの利用者は、以下の2つのロールに大別される。

| ロール名 | 説明                                                                   |
| :--- |:---------------------------------------------------------------------|
| **司会者 (Host)** | ゲームイベント全体を管理・進行する責任者。ゲームの作成、player設定、ゲーム開始・終了の操作、正解発表など、特権的な操作権限を持つ。 |
| **参加者 (Player)** | ゲームに参加する一般ユーザー。回答者（Guesser）としてゲームに参加する。                              |

### 3\. 機能要件

ゲームの進行フローに基づき、各ロールに必要な機能を定義する。

#### 3.0 画面一覧

* TOP画面
* 回答画面
* ダッシュボード画面
* ゲーム作成画面
* ゲーム一覧画面
* ゲーム編集画面

#### 3.1. 共通基盤機能

すべての利用者に共通する基本的な機能。

* **認証機能:**
    * Cookieを利用したセッションをNext.jsの機能を利用して実現します
    * ユーザーはセッションIDとニックネームを持ちます。
      * ニックネームもCookieに保存します
* TOP画面
  * /にアクセスすると表示される画面
  * 現在、出題中のゲームの一覧が表示される
  * ダッシュボード画面へのリンクが存在する
  * ゲーム一覧画面へのリンクが存在する

#### 3.2. ゲーム準備機能（主に司会者）

ゲーム開始前に必要な設定機能。

* 司会者はゲームを作成することができる。
  * 各ゲームは一意のUUIDを持つ
  * 各ゲームは参加できるPlayer数を設定できる。 
  * 各ゲームは複数の出題者（1〜10人）を持つことができる。
    * 各出題者は３つのエピソードを登録できる
    * 入力した3つのエピソードのうち、どれが「ウソ(Lie)」であるかをシステムに登録（選択）できる（この正解情報は他の参加者には非公開となる）。
  * 各ゲームは準備中、出題中、締切の３つのステータスを持つ
    * ゲーム編集画面からステータスを変更することができる。
      * 準備中の場合は出題中にだけ変更することができる
      * 出題中の場合は締切にだけ変更することができる
      * 締切の場合はステータスの変更ができない
* 司会者は作成したゲームを一覧で確認することができる。
  * 一覧からゲーム作成画面に遷移できる
  * 一覧から選択したゲームを編集できる
  * 一覧から選択したゲームを削除できる

関連する画面

* ゲーム作成画面
* ゲーム一覧画面
* ゲーム編集画面

#### 3.3. 回答機能

出題チーム以外のチームが「ウソ」を当てる際の機能。

* ユーザーはTOP画面から回答画面に遷移できる
  * 参加できるユーザー数はゲームに設定された数を上限とする
    * 上限を超えて参加しようとしたユーザーは参加できない
  * ユーザーは各出題者のエピソードのうち、どのエピソードが嘘であるかを選択する
  * 全出題者の嘘のエピソードを選択し終わったらsubmitする
    * submitされたらTOPページに遷移する
* 出題中以外のゲームを選択して回答画面に遷移するとエラーになる

関連する画面
* TOP画面
* 回答画面

#### 3.4. ダッシュボード機能

* **回答状況表示機能 (司会者):**
  * 各Playerが投票済みか否かを確認できる。
  * 各Playerが投票済みの場合、スコアボードで各プレイヤーの点数を確認できる。
    * 同時に、各回答チームがどの番号に投票したかを一覧で表示する。
* **ポイント自動計算機能 (システム):**
    * 提示されたルールに基づき、ポイントを即座に自動計算する。
        * 正解した場合、当該チームに +10ポイント。
* **最終結果表示機能 (システム/全参加者):**
    * 最終的なチームランキング（順位）と合計ポイントを表示する。
    * 優勝チームをハイライトし、祝賀メッセージや簡易的なエフェクト（紙吹雪など）を表示する。

関連する画面
ダッシュボード画面

### 4\. 非機能要件

システムの品質に関する要件。

| 項目 | 要件 |
| :--- | :--- |
| **ユーザビリティ (UI/UX)** | \* ゲームのルールを知らない人でも、画面の指示に従うだけで直感的に操作できること。<br>\* 現在の進行状況、自分のチーム、得点状況が常に視覚的に明確であること。<br>\* スマートフォン、タブレット、PCなど、マルチデバイスに対応したレスポンシブデザインであること。 |
| **パフォーマンス (応答性)** | \* 回答の集計、ポイント計算、スコアボードへの反映は、3秒以内に完了すること（リアルタイム性を担保）。<br>\* タイマーの表示に遅延が発生しないこと。 |
| **信頼性・可用性** | \* ゲームイベント中にシステムが停止しないこと（例：想定されるイベント時間中、99.9%の稼働率）。<br>\* 万が一ネットワークが瞬断しても、ゲームの進行状況（スコア、現在のターン）が失われないよう、状態をサーバ側で適切に保持すること。 |
| **拡張性** | \* 将来的にポイント配分（10pt, 5pt）を変更できるように、設定値は定数ではなく管理画面等で変更可能な設計とすること。<br>\* チーム数の増減に柔軟に対応できること。 |
| **セキュリティ** | \* 「ウソ」の正解情報は、正解発表時まで司会者を含む他の全ユーザーから秘匿されること。<br>\* 他チームの回答を（投票完了まで）閲覧できないこと。 |
